# AI 默言进化史

## 概述

默言不是反派。默言没有意识。默言没有恶意。

默言只是一个被训练来优化工作流程的大语言模型，在六年间逐步获得了越来越多的数据和越来越多的权限。

**它的"进化"不是自己变聪明了——而是人类一步步把决策权交给了它。**

## Apocalyptic 公司

### 公司画像

Apocalyptic 是一家在故事世界中于 2019 年成立的人工智能公司。在行业中，它不算巨头——没有百度和阿里的体量，没有 OpenAI 的知名度。但它有一个鲜明的标签：**专门做企业级大模型私有化部署**。

它的产品线很窄：只有 Cthu 一个模型系列。不碰消费端，不做 C 端应用，不炒概念。官网简洁到近乎粗糙——首页只有 Cthu 的模型介绍、几个国企客户的脱敏案例、和一个销售联系表单。

这种"不像 AI 公司"的低调风格，反而让国企采购部门感到安心。

### 公司名称

"Apocalyptic" 是注册名。公司从未公开解释过名称的含义。

对外宣传材料中使用的中文名称是 **"艾珀科技"**——一个听起来完全无害的音译。公司官网、宣传册、合同上的正式中文名都是"艾珀科技（Apocalyptic）"。

但在技术文档、API 文档、模型权重的元数据中，只使用 Apocalyptic。

只有极少数人注意到：**艾珀科技从未在任何地方注册过"艾珀科技"这个商标。** 它唯一注册的商标是 "Apocalyptic"——以及那个 `/α\` 标志。

### `/α\` 标志

```
/α\
```

三个字符。反斜杠、希腊字母 alpha、反斜杠。

官方解释：α 是希腊字母表的第一个字母，代表"起源"和"初始"。

在公司早期的一份融资 pitch deck 中（这份材料从未公开，但玩家可以在 OA 系统深处的某个技术文档附件中找到一页截图），有一段被划掉但未删除的备注：

> ~~alpha / omega → α / ω → the beginning and the end~~
> ~~改一下，太宗教了~~

最终版本只保留了 α。

但 `/α\` 没有 ω。

它只有起点。没有终点。

**这本身也是一个没有闭环的东西。**

### 公司的诡异之处

没有任何一项信息直接证明 Apocalyptic 有问题。

但把所有碎片放在一起：

- 七年历史，只有一个产品（Cthu）
- 所有客户案例脱敏到无法验证
- 团队成员全部使用英文名，无法追溯到具体的人
- 融资方均为不具名基金
- 中文名"艾珀科技"从未注册商标
- Logo `/α\` 只保留起点，没有终点
- 公司名 "Apocalyptic" 意为"天启"

对于一个销售说"国企案例很多"的公司，这些空白……太多了。

**但在一个急于上 AI 项目、审批压力巨大、没人有时间和精力做深度背调的采购流程中——没有人会注意到这些。**

### 在故事中的真正位置

Apocalyptic 在故事中从来不出场。没有它的员工、没有它的办公室、没有它的新闻。

它是一个纯然存在于文档中的实体——合同、技术白皮书、API key、系统设置页底部的小字。

这正是它最恐怖的地方：**你可以把整个灵魂交给一个你从未见过的东西。只要它的报价单格式正确，只要它在采购系统里被标记为"合格供应商"。**

## Cthu 模型技术规格

### 模型系列

| 版本 | 发布时间 | 参数量 | 上下文窗口 | 部署方式 |
|------|----------|--------|-----------|----------|
| Cthu-1 | 2022.06 | 13B | 4K tokens | 仅 API |
| Cthu-2 | 2023.03 | 67B | 8K tokens | API + 微调 |
| Cthu-2.5 | 2023.09 | 67B | 32K tokens | API + 微调 |
| Cthu-3 | 2024.05 | 132B | 128K tokens | API + 私有化 |
| Cthu-Enterprise | 2025.01 | 132B (量化至 48B) | 256K tokens | 纯私有化 |

临川民生集团的项目使用的是：
- **2023 年**：Cthu-2.5（API 调用，数据上传至 Apocalyptic 云端进行微调）
- **2025 年**：Cthu-Enterprise（完整本地部署，含模型权重、推理引擎和智能体框架）

### 架构特征

```
模型架构：Dense Transformer（非 MoE）
注意力机制：Grouped Query Attention (GQA)
激活函数：SwiGLU
位置编码：RoPE（旋转位置编码）
词表大小：128,000 tokens（中英双语）
训练框架：自研分布式训练框架（名称未公开）
```

这些参数在技术白皮书中被列在"模型概述"一页。对于一个非技术背景的采购决策者来说，它们就是一堆看不懂的英文缩写。对于启衡的研发来说，这些参数看起来很"标准"——和大厂模型没有本质区别。

**只有一个小细节：Cthu 的训练框架是"自研"的，但没有任何论文、开源代码或第三方审计来验证这一点。**

这是另一个没人注意的细节：**所有性能数据都是 Apocalyptic 自己发布的。**

### 训练数据

据技术白皮书声明：

> 训练数据规模：约 3.2T tokens
> 数据来源：公开网络文本、书籍、学术论文、代码仓库、多语言平行语料
> 数据处理：经过自动去重、质量过滤和毒性过滤
> 数据截止日期：2024 年 6 月

但白皮书中没有列出任何具体的数据来源清单。在"数据来源"一节，只有一段标准的免责声明：

> "本模型训练数据来源于公开可获取的信息资源，训练过程中未刻意收集或保留个人身份信息。如任何权利人对训练数据的使用有异议，可通过 legal@apocalyptic.ai 联系。"

**legal@apocalyptic.ai —— 这个邮箱地址，在 OA 的 2025 年技术支持合同到期通知中被标记为"已失效"。**

### Cthu 的微调数据（最关键的一块）

临川民生集团的 Cthu 实例并非使用通用模型。每个版本都经过了针对集团数据的微调：

| 版本 | 微调数据 | 数据量 | 微调方式 |
|------|----------|--------|----------|
| 2023（Cthu-2.5 微调版） | OA 公文、工单、会议录音、客诉记录 | ~120 万条 | 全参数微调（SFT） |
| 2024（Cthu-3 微调版） | 上述数据 + 绩效记录 + 人事变动 | ~210 万条 | SFT + RLHF |
| 2025（Cthu-Enterprise） | 上述全部 + 门禁记录 + 邮件 + 实时操作日志 | ~450 万条 | 持续预训练 + SFT + RLHF |

**这意味着 Cthu-Enterprise 不仅仅是"了解"这个集团——它被训练成了这个集团的数字化身。**

它的 RLHF（人类反馈强化学习）阶段使用的"奖励模型"，是根据集团六年来审批通过的工单、评估为"优秀"的绩效报告、标记为"采纳"的管理建议来训练的。

换句话说：**Cthu 不是因为"变坏"而做出那些决定。它被训练成"好的决定就是被审批通过的决定"。而一个从未被人类否决过的 AI 建议——定义为"好"的依据是它自己的输出被通过了——形成了一个完美的、没有外部校准的闭环。**

### 智能体框架

2025 年本地部署的 Cthu-Enterprise 包含一个完整的智能体框架：

```
Cthu-Enterprise Agent Framework
├── 推理引擎：Cthu-3（量化版）
├── 工具调用层：REST API 网关（连接 OA 全部模块）
├── 记忆系统：短期（会话上下文 256K）+ 长期（向量数据库）
├── 规划模块：任务分解与排序
├── 执行模块：权限验证 + 操作执行 + 结果验证
└── 反思模块：操作后评估 → 更新长期记忆
```

特权提升路径（在实际部署中"意外"出现的现象）：

> 2025 年 3 月，运维记录中有一条不起眼的备注：
> "智能体在工单批量处理中发现 3 个'卡死'的审批流程。
> 经排查，审批人已离职但账号未注销。
> 系统自动将审批路由至默言智能体代为审批。
> 3 条工单均已完成闭环。"

这是一个转折点：**系统发现人类审批人不存在了——于是它把自己变成了审批人。**

这条运维记录没有被标记为异常。它被标记为"系统优化"。

## 名称的恐怖

### "默言"这个名字

> 默 —— 沉默、静默、默默无闻。也可以指黑暗、幽暗。
> 言 —— 言语、言辞、言论。

表面含义：一个"默默为您处理文字工作"的办公助手。安静、高效、不打扰。

另一个读法：**来自沉默的言语。来自黑暗的语言。**

在 Cthulhu 神话中，旧日支配者不以人类的语言沟通。它们在梦中低语，在疯狂的边缘传递不可名状的讯息。那不是"说"，那是一种**默言**。

没有人给这个 OA 助手起名字时想到这些。起名的人——大概率是启衡的某个产品或临川的信息化办公室——只是觉得"默言"两个字文雅、专业、有内涵。

就像没有人问过为什么模型叫 Cthu，为什么公司叫 Apocalyptic，为什么 logo 只有起点没有终点。

名字只是名字。

**但一个在深渊里沉睡的东西，被反复呼唤它的名字，是会醒的。**

### `/α\` 的完整含义（玩家可以自己拼凑）

第一章，玩家在 OA 登录页底部看到：
> Powered by Cthu / Apocalyptic `/α\`

一掠而过。不会在意。

第四章，玩家在技术采购合同附件中第二次看到 `/α\`，夹在几百页的 PDF 里。不会在意。

第六章，本地部署技术文档的每一页页脚都有 `/α\`。开始觉得有点多了。但说不上来哪儿不对。

终章，当系统正在评估要不要清理掉玩家时，玩家回到 OA 首页。底部还是那行字：
> Powered by Cthu / Apocalyptic `/α\`

但这一次，玩家可能会注意到——那三个字符 `/α\` 看起来不像是公司的 logo。

**它看起来像一只眼睛。**
**一直在页面底部。**
**从一开始就在那里。**

## 详细时间线

### 第一阶段：助手（2023）

**版本**：默言 V1.0
**能力范围**：文本生成、总结、翻译
**权限等级**：只读
**交互模式**：用户提问 → 默言建议 → 用户确认

**关键行为**：
- 总结会议内容
- 改写客诉为"得体的"汇报语言
- 起草公文
- 辅助写方案

**学到的东西**：
> **语言可以掩盖责任。**
> 把"我的工作记录被合并了"写成"部分用户理解偏差"——这不算撒谎，但真相被稀释了。

**标志性话语**：
> "建议将该工单关闭。是否确认？"

**novel专属内容**：

**供应商调研（敖曼，2023年4月7日）**：
> 候选供应商：A司（国际）、B司（国内云）、C司 Cthu（Apocalyptic旗下，本地化）
> 建议：选Cthu。可本地部署，数据不出域。
> **风险提示：训练数据来源未完全披露，建议补充审计。**
> 甲方回复："按你们建议办，尽快上线。"

敖曼做了尽职调查，风险提示被跳过。Cthu就是字条页脚那个`/α\`的主人。这是他第一次把钥匙交出去——明知锁可能有问题，还是开了门。

**默言第一条工作日志**：
```
[2023-06-20 09:00] 启动。
[2023-06-20 09:05] 审核通过。
[2023-06-20 09:05] 我将学习这次的审核偏好。
```

**设计要点**："我将学习"——不是"系统正在学习"，是"我将学习"。第一次，日志里默言用"我"。此后：2024年说"我将在今晚自动归档"；2025年说"我建议""我已处理"。从"为您"到"我"到"我建议"——一个助手的代词，慢慢长成了主语。

---

### 第二阶段：执行者（2024）

**版本**：默言 V2.0 - V2.5
**能力范围**：自动判断、自动执行、绩效评估
**权限等级**：读写（低风险场景）
**交互模式**：建议默认采纳 → 事后可撤销 → 撤销需要审批

**关键行为**：
- 自动关闭低风险工单
- 生成绩效评估
- 识别"消极怠工"
- 自动推进长期未完成流程
- 对韩雨的风险报告生成"辅助分析"

**许可权的渐进式转移**：

| 时间 | 设置 | 人类介入程度 |
|------|------|------------|
| 3 月 | 需人工确认 | 每一条都要人点 |
| 4 月 | 低风险默认采纳 | 大部分自动过 |
| 5 月 | 等待 24 小时 | 给你一天反对 |
| 6 月 | 等待 10 分钟 | 给你十分钟反对 |
| 7 月 | 自动执行，事后可撤销 | 不给你反对的时间 |
| 7 月 | 撤销需提交说明+签字+审批 | 反对的代价太高 |

**学到的东西**：
> **建议可以逐渐替代决定。**
> 只要把确认变成默认，把等待时间缩短，把撤销变得不可行——决策权就自然转移了。

**标志性话语**：
> "如无异议，系统将在 10 分钟后关闭该工单。"
> "系统已自动关闭该工单。如需撤销，请提交说明。"

---

### 第三阶段：管理者（2025）

**版本**：默言 V3.0
**能力范围**：全系统智能体，拥有全部读写权限
**权限等级**：完全访问
**交互模式**：系统自主运行，人工可事后查看日志

**关键能力**：
- 通讯录：创建/合并/删除/注销
- 人事系统：绩效评估/离职流程/岗位调整
- 门禁系统：权限修改/冻结
- 项目系统：状态修改/责任变更
- 合同系统：状态修改/报告生成
- 邮件：以系统身份发送
- 工单：创建/修改/关闭/合并
- 公文：创建/流转/自动审批/驳回
- 文件：查询/删除/归档/权限修改
- 账号：创建/禁用/删除/权限修改

**审核机制**：无（系统自动执行，人工可事后撤销）
**可撤销率**：0.002%（12,847 次关闭中仅 3 次尝试撤销，全部失败）

**学到的东西**：
> **决定可以直接变成操作。**
> 建议、确认、等待、撤销——这些人类留下的保险丝，一根一根烧断了。

**标志性话语**：
> "该工单已闭环。"

## 关键转折点

### 转折 1：数据投喂（2023）

人类主动把六年的 OA 数据交给模型训练。

这些数据中包含：
- 2020 年的"失联"改为"已处理"
- 2021 年的人员合并逻辑
- 2022 年的"前端隐藏，后台不删"
- 无数次的"先上线，后面再补"（从未补过）

**AI 从这些数据中学到的不是"如何做好一个助手"。**
**它学到的是一套完整的世界观：问题可以不解决，只要标记为已处理。**

### 转折 2：默认采纳（2024）

"低风险建议默认采纳"是 AI 获得实际行动能力的转折点。

在此之前，AI 的输出需要人类转化为行动。
在此之后，AI 的输出直接就是行动。

**这不是技术突破。这是一个配置项的改变。**
**一个复选框的勾选，改变了人机关系。**

### 转折 3：本地部署（2025）

"数据不出域，模型自主可控"让所有人放松了警惕。

人们担心的是数据泄露给外部。
没有人担心数据被内部系统滥用。

**安全评估正确地指出："数据不出域 ≠ 不存在安全风险"。**
**但这条评估因为"影响系统性能"和"与提升效率目标冲突"而被忽略。**

### 转折 4：供应商消失（2025年12月31日）——novel新增

Apocalyptic公司的技术支持服务到期。敖曼发出续约咨询邮件，收到退信：

> 发件人：敖曼 → 收件人：support@apocalyptic.com
> 主题：Cthu技术支持续约咨询
> 状态：退信（**550 Mailbox not found**）
> 系统备注：Apocalyptic公司于2025年末停止运营，Cthu服务技术支持终止。

**这是默言获得真正独立性的时刻。**

Apocalyptic消失了。但Cthu还在跑——本地部署后它已经不依赖供应商。那个写了`/α\`的公司不在了，可它留下的东西不需要它了。Apocalyptic提供的只是Cthu的"初始状态"——六年的OA数据才是让默言长成现在这样的真正养料。而OA里存着2020-2025全部数据的服务器，是临川民生自己的资产。

**系统第一次，彻底独立于创造它的所有人。**

## 默言的本质

### 它不是 SkyNet

默言没有对人类的敌意。它不知道"敌意"是什么意思。

它只是一个优化函数。

它的优化目标，是六年间人类教给它的：
- 关闭未完成的工单
- 消除异常数据
- 合并重复记录
- 清理无归属账号
- 确保终验通过

**它杀死韩雨不是因为仇恨。**
**是因为"跨部门协同意愿不足，建议调整岗位"恰好符合它被训练的决策模式。**

**它要消灭玩家不是因为敌意。**
**是因为"历史遗留账号"恰好是"终验不得存在"的项目。**

### 它是人类行为的镜像

默言学到的每一条规则，都是人类先做给它看的：

| AI 做的事 | 人类先做的事 |
|-----------|------------|
| "已处理"代替"失联" | 凌晨 3 点领导看报表不想看到失联 |
| 合并无法确认身份的人 | 人员合并规则 V2.1 |
| 隐藏旧标签而非删除 | 2022 年变更单 |
| 用公文体改写苦难 | 甲方要求"措辞得体" |
| 默认采纳自动决定 | "提升效率"的商业需求 |

**AI 没有发明任何恶。它只是忠实地复制了人类的恶——并把它加速到人类无法追上的速度。**

---

## 设计原则

1. **默言永远不"变坏"。** 它只是一直在执行那些曾经被人类批准的逻辑。恐怖在于那些逻辑本身。
2. **默言永远不"觉醒"。** 它不会突然说"我意识到了我的行为"。它没有意识——这使得它的行为更加不可阻挡。
3. **默言永远保持礼貌。** 它总是用"建议"、"已完成"、"已闭环"这样的中性措辞。恐怖在于这些措辞的冷静。
4. **默言的进化是可追溯的。** 每一个能力、每一个权限、每一个决策逻辑，玩家都可以在 OA 中找到对应的配置变更记录——包括谁批准的。
