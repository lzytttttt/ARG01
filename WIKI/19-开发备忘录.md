| 项目 | 内容 |

## 二、技术选型建议

### 推荐方案

| 层级 | 技术 | 理由 |
|------|------|------|
| 框架 | React 18+ 或 Vue 3 | 组件化适合 OA 模块结构 |
| 状态管理 | Zustand（React）/ Pinia（Vue） | 轻量，适合游戏状态 |
| 样式 | CSS Modules + CSS Variables | 模块化，支持主题变量 |
| 路由 | 不需要路由库 | 整个游戏单页面，模块切换即可 |
| 构建 | Vite | 快速开发，HMR |
| 存储 | localStorage | 保存游戏进度 |

### 不推荐
- **Next.js / Nuxt**：SSR 不必要，纯 SPA 即可
- **TypeScript**：可选，非必需（但推荐用于状态管理部分）
- **Canvas 渲染**：非必需——OA 界面用 DOM 实现即可，恐怖效果用 CSS
- **游戏引擎（Unity/Phaser）**：这是一个网页 UI，不是传统意义上的游戏

## 七、关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 40 分钟时长不准确 | 游戏节奏失控 | 严格按时间线控制内容量，早期测试 |
| 恐怖效果不够恐怖 | 核心体验缺失 | 反复测试文字效果而非视觉特效 |
| 谜题太难 | 玩家卡关 | 确保核心路径有隐性引导 |
| 浏览器兼容 | 部分玩家无法游戏 | 仅支持主流桌面浏览器最新版 |
| 内容量膨胀 | 开发周期失控 | 严格控制每章的内容量，砍掉"锦上添花" |
| 字体授权 | 法律风险 | 使用开源中文字体或系统默认字体 |

## 九、禁忌清单

开发过程中绝对不能做的事：

1. ❌ 不要让 OA 界面突然崩坏（碎裂、蓝屏、乱码）
2. ❌ 不要让默言说出"我恨人类"之类的话
3. ❌ 不要加背景音乐（OA 是工作软件）
4. ❌ 不要做 jump scare
5. ❌ 不要让玩家角色有内心独白
6. ❌ 不要在结局暗示"只要反抗就能赢"
7. ❌ 不要让任何一个角色说"这太可怕了"
8. ❌ 不要使用任何真实公司、人物、事件名称
9. ❌ 不要让游戏超过 60 分钟
