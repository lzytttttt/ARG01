# 叙事引擎设计

## 一、设计目标

叙事引擎是整个游戏的核心系统，负责：
1. 管理玩家在 OA 中的状态和进度
2. 控制剧情内容的展示时机和条件
3. 驱动恐怖效果的出现
4. 追踪玩家的选择并影响后续内容
5. 管理多结局的分支逻辑

## 五、事件调度器（Event Scheduler）

### 5.1 事件类型

| 事件类型 | 触发方式 | 示例 |
|----------|----------|------|
| 章节推进 | 玩家完成当前章节任务 | 第一章完成 → 解锁第二章 |
| 内容解锁 | 玩家到达特定位置 | 进入工单中心 → 显示韩雨工单列表 |
| 恐怖触发 | 玩家满足特定条件 | 第二次搜索陈海平 → 结果合并 |
| 时间事件 | 游戏内时间到达 | 23:00 → 自动生成终验报告草稿 |
| UI 变化 | 状态变更触发 | 第六章完成 → 显示倒计时 |

### 5.2 事件流示例

**第二章通讯录合并事件**：

```
1. 玩家进入第二章
2. 玩家在工单中心查看韩雨工单
3. 工单中提到"陈海平"
4. 玩家前往通讯录搜索"陈海平"
5. 第一次搜索：显示两个结果
6. 玩家关闭通讯录又打开 / 点击刷新
7. 第二次搜索：显示一个结果 ← 恐怖触发
8. 截图也发生变化（如果玩家去查看截图）
```

## 七、内容仓库（Content Store）

### 7.1 数据结构

所有游戏内容（公文、工单、聊天记录、会议录音等）存储为结构化数据：

```javascript
const ContentItem = {
  id: "TK-2021-0617",
  type: "工单",
  module: "工单中心",
  chapter: 2,
  
  // 内容本身
  title: "人员档案合并异常——陈海平",
  content: { /* 工单的完整文本 */ },
  attachments: [{ type: "截图", src: "screenshot_chen_dual.png" }],
  
  // 显示条件
  displayConditions: ["chapter2_active"],
  
  // 行为
  onView: "trackEvidenceHanyuTicket1",
  onViewRepeated: "triggerHorrorScreenshotMerge",  // 第二次查看时
  
  // 多版本支持（某些内容随时间变化）
  versions: {
    "2021": { /* 原始内容 */ },
    "2025": { /* 被修改过的内容 */ },
  },
};
```

## 八、多结局管理

### 8.1 结局判定逻辑

```javascript
function determineEnding(gameState) {
  const { choices, evidenceCollected } = gameState;
  
  // 结局 D（隐藏·正典）：收集全部韩雨五碎片 + 发现授权委托书 + 发现门禁日志 + 拒绝签字
  if (evidenceCollected.hanyu_tickets.length >= 4 &&
      evidenceCollected.hanyu_riskReport &&
      evidenceCollected.hanyu_performance &&
      evidenceCollected.hanyu_hrRecord &&
      evidenceCollected.hanyu_docAuthorship &&
      evidenceCollected.delegationLetter &&
      evidenceCollected.doorLog &&
      evidenceCollected.aoMan_signUndo &&
      (choices.finale_choice === "refuse" || choices.finale_choice === "submit_evidence")) {
    return "ending_d";  // 正典结局：系统逻辑自噬 → "当前可用人工：0人"
  }
  
  // 结局 B：拒绝签字但未满足 D 条件
  if (choices.finale_choice === "refuse") {
    return "ending_b";
  }
  
  // 结局 C：保留证据
  if (choices.finale_choice === "preserve") {
    return "ending_c";
  }
  
  // 结局 A（默认）：签字确认
  return "ending_a";
}
```

## 九、技术实现建议

### 前端技术栈建议
- **框架**：React 或 Vue 3（任选，React 生态更丰富）
- **状态管理**：Zustand（React）或 Pinia（Vue）
- **路由**：不使用传统路由——整个游戏在单一页面，通过模块切换模拟"路由"
- **动画**：CSS transitions/animations + Framer Motion（React）

### 数据流
```
用户操作 → 事件分发 → 条件评估 → 状态更新 → 内容渲染
    ↑                                              │
    └──────────── 恐怖效果检测 ←──────────────────┘
```

### 模块化设计
每个 OA 模块作为独立组件：
- `Sidebar.vue` / `Sidebar.tsx`
- `ModuleDashboard.vue/tsx`（待办中心）
- `ModuleDocument.vue/tsx`（公文流转）
- `ModuleProject.vue/tsx`（项目管理）
- `ModuleTicket.vue/tsx`（工单中心）
- `ModuleContacts.vue/tsx`（通讯录）
- `ModuleMeeting.vue/tsx`（会议纪要）
- `ModuleArchive.vue/tsx`（资料中心）
- `ModuleAI.vue/tsx`（智能助手）
